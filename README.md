# Frontend TSZH System

Frontend реализует пользовательский интерфейс веб-системы ТСЖ. Приложение работает с backend через REST API и использует JWT-токен из `localStorage`.

## Стек

- React
- Vite
- React Router
- Material UI
- Axios
- jwt-decode

## Запуск

1. Установить зависимости:

```powershell
cd D:\Python_project\dimas\main_frontend
npm install
```

2. Запустить dev-сервер:

```powershell
npm.cmd run dev -- --host 127.0.0.1
```

Frontend будет доступен по адресу:

```text
http://127.0.0.1:5173/
```

Backend должен быть запущен на:

```text
http://127.0.0.1:8000
```

Базовый URL API находится в:

```text
src/api/api.js
```

## Основная логика приложения

Приложение предназначено для работы жильцов, диспетчеров, администраторов и исполнителей в системе ТСЖ.

После входа пользователь получает JWT-токен. Токен сохраняется в `localStorage`. На основе роли из токена frontend показывает доступные страницы.

## Роли

### Жилец

Жилец видит:

- главную страницу;
- создание заявки;
- профиль;
- детали заявки.

Жилец может:

- создавать заявки только по подтвержденным адресам;
- смотреть свои заявки;
- добавлять комментарии;
- управлять контактными данными;
- добавлять, редактировать, удалять адреса;
- выбирать основной адрес.

### Исполнитель

Исполнитель пока использует такой же интерфейс, как жилец:

- главная;
- создание заявки;
- профиль;
- детали заявки.

### Диспетчер

Диспетчер видит:

- главную страницу;
- панель диспетчера;
- профиль;
- детали заявки.

Диспетчер может:

- просматривать все заявки;
- менять статус заявок;
- подтверждать адреса жильцов.

Диспетчер не видит страницу создания заявки и не может создавать заявки через backend.

### Администратор

Администратор видит только страницу:

- пользователи.

Администратор может:

- смотреть список пользователей;
- назначать роли другим пользователям;
- подтверждать адреса жильцов.

Администратор не видит:

- главную;
- создание заявки;
- панель диспетчера;
- профиль.

## Маршруты frontend

`/`

Страница входа. Если пользователь уже авторизован, выполняется редирект:

- `admin` -> `/users`
- `dispatcher` -> `/dispatcher`
- остальные роли -> `/dashboard`

`/register`

Регистрация жильца.

`/dashboard`

Главная страница со списком заявок пользователя и счетчиками по статусам.

Доступ:

- `resident`
- `executor`
- `dispatcher`

`/create-ticket`

Создание заявки.

Доступ:

- `resident`
- `executor`

`/dispatcher`

Панель диспетчера.

Доступ:

- `dispatcher`

`/users`

Страница администратора для управления пользователями и подтверждения адресов.

Доступ:

- `admin`

`/tickets/:id`

Комментарии и детали заявки.

Доступ:

- `resident`
- `executor`
- `dispatcher`

`/profile`

Личный кабинет.

Доступ:

- `resident`
- `executor`
- `dispatcher`

## Страницы

### LoginPage

Форма входа.

Пользователь вводит:

- email;
- пароль.

После успешного входа токен сохраняется в `localStorage`, а пользователь перенаправляется на страницу, соответствующую его роли.

### RegisterPage

Форма регистрации жильца.

Поля:

- ФИО;
- телефон;
- email;
- пароль;
- улица;
- дом;
- квартира;
- лицевой счет.

После регистрации создается пользователь и первый адрес. Адрес получает статус `Не подтвержден`, поэтому заявки по нему можно создавать только после подтверждения диспетчером или администратором.

### DashboardPage

Показывает заявки текущего пользователя:

- новые;
- в работе;
- завершенные.

Для каждой заявки отображаются:

- номер;
- описание;
- адрес;
- статус;
- приоритет;
- кнопка открытия деталей.

### CreateTicketPage

Страница создания заявки.

Логика адресов:

- frontend загружает только подтвержденные адреса через `/addresses/my/verified`;
- если подтвержденный адрес один, он автоматически подставляется и не редактируется;
- если подтвержденных адресов несколько, показывается выпадающий список;
- ручной ввод адреса запрещен;
- если подтвержденных адресов нет, форма создания заявки блокируется.

При отправке передается:

- описание;
- `category_id`;
- `address_id`.

### ProfilePage

Личный кабинет состоит из двух частей.

Контактные данные:

- ФИО;
- email;
- телефон;
- пароль.

Пользователь видит данные в виде карточек, выбирает нужное поле, редактирует его и сохраняет изменение.

Адреса:

- список всех адресов пользователя;
- статус подтверждения;
- признак основного адреса;
- лицевой счет;
- добавление адреса;
- редактирование адреса;
- удаление адреса;
- выбор основного адреса.

После редактирования адрес становится неподтвержденным.

Удаление адреса может быть запрещено backend-ом, если по адресу есть активные заявки.

### DispatcherPage

Панель диспетчера состоит из двух блоков.

Адреса на подтверждение:

- адрес;
- лицевой счет;
- пользователь;
- кнопка подтверждения.

Заявки:

- список всех заявок;
- описание;
- адрес;
- статус;
- кнопки изменения статуса.

### UsersPage

Административная страница.

Содержит:

- адреса на подтверждение;
- список пользователей;
- выбор роли для каждого пользователя.

Администратор не может изменить роль самому себе.

## Навигация

Навигация формируется по роли из JWT:

- `admin` видит только `Пользователи`;
- `dispatcher` видит `Главная`, `Диспетчер`, `Профиль`;
- `resident` и `executor` видят `Главная`, `Создать заявку`, `Профиль`.

В шапке отображается бейдж текущей роли:

- Жилец;
- Диспетчер;
- Администратор;
- Исполнитель.

## Работа с API

Axios-инстанс находится в:

```text
src/api/api.js
```

Текущий base URL:

```js
http://127.0.0.1:8000
```

Токен прикладывается вручную в защищенных запросах через заголовок:

```http
Authorization: Bearer <token>
```

## Сборка

```powershell
npm.cmd run build
```

При сборке Vite может показать предупреждение о размере чанка больше 500 kB. Это не является ошибкой и не мешает работе приложения.

## Ограничения текущей версии

- Нет автоматического refresh token.
- Токен хранится в `localStorage`.
- Нет глобального Axios interceptor.
- Механизм связанных обращений пока не отображается в интерфейсе.
- Подтверждение адреса выполняется вручную диспетчером или администратором.
